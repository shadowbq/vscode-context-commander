# Developer Guide — Context Commander

This guide covers the full development lifecycle: local setup, iterative development, static analysis, packaging, installing a local build, and publishing to GitHub and the VS Code Marketplace.

---

## Prerequisites

| Tool | Minimum version | Notes |
|------|----------------|-------|
| Node.js | 25.x | Use the version specified in the CI workflow |
| npm | 11.x | Ships with Node 25 |
| VS Code | 1.85.0 | Required to run the Extension Development Host |
| `@vscode/vsce` | 3.9.x | Installed as a dev dependency — no global install needed |

---

## Initial Setup

```bash
git clone https://github.com/shadowbq/vscode-context-commander.git
cd vscode-context-commander
npm ci
```

Use `npm ci` (not `npm install`) to get a reproducible install that respects `package-lock.json`.

## Project Structure

```
src/                   TypeScript source files
  extension.ts         Entry point — activate/deactivate, command registration
  types.ts             Shared interfaces (Binding, WhenClause, …)
  commandRunner.ts     Sequential command execution + variable substitution
  whenClause.ts        File-matching logic (ext / languageId / glob / isDirectory)
  slotManager.ts       Config → context variable sync, slot assignments
  settingsPanel.ts     WebView panel for the visual settings editor
media/
  settings.js          Webview-side JavaScript for the settings panel
out/                   Compiled output (git-ignored, generated by build scripts)
.github/workflows/
  publish.yml          CI: type-check → lint → build → vsce package → publish
```

## Development Workflow

### Watch mode (incremental rebuild)

```bash
npm run watch
```

Runs esbuild in watch mode with source maps. Rebuild is near-instant on each save.

### Running in the Extension Development Host

1. Open the repo folder in VS Code.
2. Press **F5** (or **Run → Start Debugging**). The `.vscode/launch.json` configuration targets `extensionHost` — no specific file needs to be selected first. VS Code compiles the extension then opens a new **Extension Development Host** window with it loaded.
3. In the Extension Development Host window, open a folder so the Explorer sidebar is active.
4. Right-click a file to test context menu entries.

> **Tip:** After editing source files and saving, the watch task rebuilds automatically. Press **Ctrl+Shift+F5** (Restart) in the host window to reload the extension without relaunching the entire host.

You can also use the **Developer: Reload Window** command (`Ctrl+Shift+P → Reload Window`) in the host window to pick up changes.

### Iterating on the WebView settings panel

Open the Settings panel via the Command Palette: **Context Commander: Open Settings**.  
Changes to `media/settings.js` take effect on the next panel open (no rebuild required — it is loaded directly by the WebView runtime). Changes to `src/settingsPanel.ts` (the HTML template or message handlers) require a rebuild and extension reload.

## Static Analysis

### Type checking

```bash
npm run type-check
```

Runs `tsc --noEmit`. No output files are written — esbuild handles actual compilation. Fix all reported errors before opening a pull request.

### Linting

```bash
npm run lint
```

Runs ESLint with the `@typescript-eslint` ruleset over `src/`. The `media/` directory is excluded (plain JS, no type info).

## Building

### Development build (with source maps)

```bash
npm run compile
```

Outputs `out/extension.js` + `out/extension.js.map`. Use this during active development.

### Production build (minified, no source maps)

```bash
npm run build
```

Outputs a minified `out/extension.js`. This is what `vsce package` runs automatically via the `vscode:prepublish` hook.

---

## Packaging a `.vsix`

```bash
npm run package
```

This runs `vsce package`, which:
1. Triggers `vscode:prepublish` → `npm run build` (minified production bundle).
2. Applies `.vscodeignore` to exclude source, tests, CI, and `node_modules`.
3. Writes `context-commander-<version>.vsix` in the repo root.

---

## Installing a Local Build

### From the command line

```bash
code --install-extension context-commander-0.1.0.vsix
```

Replace the filename with the actual version output by `npm run package`.

### From the VS Code UI

1. Open the Extensions sidebar (`Ctrl+Shift+X`).
2. Click the **…** menu (top-right of the Extensions panel).
3. Choose **Install from VSIX…** and select the `.vsix` file.

Reload VS Code after installing.

### Uninstalling a locally installed build

```bash
code --uninstall-extension shadowbq.context-commander
```

## Bumping the Version

Edit `package.json` and change the `"version"` field following [Semantic Versioning](https://semver.org/):

- `patch` — bug fixes (`0.1.0` → `0.1.1`)
- `minor` — new backwards-compatible features (`0.1.0` → `0.2.0`)
- `major` — breaking changes (`0.1.0` → `1.0.0`)

Also update `CHANGELOG.md` with an entry for the new version before tagging/releasing.

## Publishing to GitHub

### Automated (recommended)

The CI workflow (`.github/workflows/publish.yml`) triggers automatically on every push to `main` that touches `package.json`. It:

1. Runs type-check and lint.
2. Produces a minified build.
3. Packages a `.vsix`.
4. Publishes to the VS Code Marketplace (requires `VSCE_PAT` secret — see below).
5. Creates a GitHub Release tagged `v<version>` with the `.vsix` as an artifact and `CHANGELOG.md` as the release body.

### Dry-run (package only, no publish or release)

Trigger the workflow manually from the **Actions** tab and check **Package only (do not publish or release)**:

```
GitHub → Actions → Publish Extension → Run workflow → dry_run: true
```

### Manual release

```bash
# 1. Commit and push your changes
git add .
git commit -m "chore: release v0.2.0"
git push origin main

# 2. Tag the release
git tag v0.2.0
git push origin v0.2.0
```

GitHub will show the tag under **Releases**. To create a formal release with the `.vsix` attached, use the GitHub UI or `gh` CLI:

```bash
gh release create v0.2.0 context-commander-0.2.0.vsix \
  --title "v0.2.0" \
  --notes-file CHANGELOG.md
```

## Publishing to the VS Code Marketplace

### One-time setup

1. Create a [Personal Access Token (PAT)](https://code.visualstudio.com/api/working-with-extensions/publishing-extension#get-a-personal-access-token) in Azure DevOps with the **Marketplace → Manage** scope.
2. Add the PAT as a repository secret named `VSCE_PAT` (Settings → Secrets and variables → Actions).
3. Ensure the publisher `shadowbq` exists at [marketplace.visualstudio.com/manage](https://marketplace.visualstudio.com/manage).

### Publish via CI (normal path)

Push a commit to `main` that bumps `version` in `package.json`. The workflow handles everything.

### Publish manually

```bash
# Authenticate once (stores token in ~/.vsce)
npx @vscode/vsce login shadowbq

# Build and publish in one step
npm run publish

# Or publish a pre-built .vsix
npx @vscode/vsce publish --packagePath context-commander-0.1.0.vsix
```

### Verify the published extension

```
https://marketplace.visualstudio.com/items?itemName=shadowbq.context-commander
```

Marketplace listings typically update within a few minutes of a successful publish.

## Testing

The project has two distinct testing layers with different tools:

| Layer | Tool | Requires Extension Host? |
|---|---|---|
| Unit (pure logic) | [Vitest](https://vitest.dev/) | No — runs in Node directly |
| Integration | [`@vscode/test-cli`](https://github.com/microsoft/vscode-test-cli) + [Extension Test Runner](https://marketplace.visualstudio.com/items?itemName=ms-vscode.extension-test-runner) | Yes — spawns a real VS Code instance |

---

### Unit Tests (Vitest)

The modules `whenClause.ts`, `commandRunner.ts`, and `slotManager.ts` contain pure logic and are covered by tests in `src/__tests__/`.

```bash
npm run test:unit
```

---

### Integration Tests (Extension Test Runner)

Integration tests run inside a real VS Code Extension Development Host. They can assert that commands are registered, context variables are set, and menus appear.

#### Prerequisites

Install the **Extension Test Runner** VS Code extension:

```
ext install ms-vscode.extension-test-runner
```

Or search for `ms-vscode.extension-test-runner` in the Extensions view.

#### Setup

```bash
npm install --save-dev @vscode/test-cli @vscode/test-electron
```

Tests live in `src/test/`. The discovery config is in `.vscode-test.mjs` at the repo root.

Add to `package.json` scripts:

```json
"test:integration": "vscode-test"
```

Add to `.vscodeignore` (source maps from integration test compilation should not ship):

```
src/test/**
out/test/**
.vscode-test.mjs
```

#### Running tests in the Terminal

```bash
npm run test:integration
```

#### Running tests inside VS Code (Extension Test Runner UI)

Once the extension is installed, a **Test Explorer** (flask icon) appears in the Activity Bar. It automatically discovers `.vscode-test.mjs` and lists all test suites. You can:

- Run individual tests or entire suites with the play button
- Debug a failing test with the debug button (sets breakpoints in the Extension Development Host)
- See inline pass/fail annotations in the editor gutter

#### Source maps (required for the Test Runner UI)

The Extension Test Runner maps errors back to TypeScript source lines via source maps. Make sure the test compilation step uses `--sourcemap`. If you add a dedicated `tsc` compile step for tests, set `"sourceMap": true` in `tsconfig.json` (the esbuild `--sourcemap` flag already handles the main bundle).

#### CI note

Integration tests require a display server. On Linux CI runners add:

```yaml
- name: Run integration tests
  run: xvfb-run -a npm run test:integration
```

## Troubleshooting

| Symptom | Likely cause | Fix |
|---------|-------------|-----|
| `out/extension.js` not found | Build hasn't run | `npm run compile` |
| Context menu entry missing | `contextCommander.hasBindings` is `false` | Add at least one binding in Settings |
| `minimatch` type error | Wrong import style | Use `import { minimatch } from 'minimatch'` (named import) |
| `String.replaceAll` type error | `lib` target too old | Ensure `"lib": ["ES2021"]` in `tsconfig.json` |
| `vsce` publish fails with 401 | PAT expired or wrong scope | Regenerate PAT with Marketplace → Manage scope |
| WebView shows blank panel | Content Security Policy mismatch | Check `_buildHtml()` CSP in `settingsPanel.ts` |
